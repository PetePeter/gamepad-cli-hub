Completion Recap — Feature Proposal

Feature Low Risk v3 — read-tracker gate  ·  gamepad-cli-hub  ·  2026-05-20

Problem Statement

The Gap

When an AI agent calls plan_complete, the hub accepts it unconditionally. There is no check that the agent actually read the plan, let alone read it recently enough to still remember what "done" means. LLMs drift — plans get closed with partial work because the agent convinced itself it was done without ever re-checking the criteria it agreed to at the start.

User POV

"I want the hub to gate plan completion on read-verification. If the agent never read the plan — call BS. If it read it ages ago and made a ton of changes since — ask it to re-read. Only if it read it recently should it be allowed to complete and get the recap. The plan might not even have AC sections — in that case, tell the LLM to go re-read the description anyway."

Core Mechanic — Three-Tier Read Gate

The hub tracks every plan_get call per (plan, session) pair — recording the timestamp and PTY write count at the moment of the read. On plan_complete, three checks run in order:

🔍
Has this session ever called plan_get for this plan?
Checked via PlanReadTracker.getRead(planId, sessionId)
❌
NO → GATE 1: "You haven't read this plan."
Write CALL-BS block to PTY. plan_complete is rejected — status stays unchanged.
✅
Is the read stale? (time elapsed > threshold AND PTY writes since read > threshold)
Stale = now − readAt > staleTimeMs AND currentWrites − writesAtRead > staleWriteCount
⚠️
YES → GATE 2: "You read this plan N minutes ago and have made significant changes since."
Write STALE-READ block to PTY. plan_complete rejected — status unchanged.
✅
Fresh read — does the plan description contain AC or Done Statement sections?
📋
YES → GATE 3 PASS: Write RECAP block with extracted criteria to PTY. plan_complete proceeds.
📄
NO SECTIONS → GATE 3 PASS (soft): Write RECAP block with re-read instruction to PTY. plan_complete proceeds.
No criteria to list, but agent is told to re-read the plan description and verify manually.

Gates 1 and 2 block completion — the plan stays in its current status. Gate 3 always allows completion but gives the LLM the recap to respond to.

PlanReadTracker — New Module

A lightweight in-memory tracker (no disk persistence needed — lost on restart is fine, since a fresh session should re-read anyway).

interface ReadRecord {
  planId:         string
  sessionId:      string
  readAt:         number   // Date.now() at moment of plan_get
  ptyWriteCount:  number   // PTY write count for this session at moment of read
}

class PlanReadTracker {
  recordRead(planId, sessionId, ptyWriteCount): void
  getRead(planId, sessionId): ReadRecord | undefined
  isStale(record, currentWriteCount, now): boolean
    // stale = (now - readAt > STALE_TIME_MS) AND (currentWrites - record.ptyWriteCount > STALE_WRITE_COUNT)
  clear(sessionId): void   // called on session close
}

Defaults: STALE_TIME_MS = 10 * 60 * 1000 (10 min) · STALE_WRITE_COUNT = 50 PTY writes

Both thresholds must be exceeded simultaneously — long time with no activity is not stale; lots of writes in 30 seconds is not stale.

PTY Write Count Tracking

PtyManager already receives every pty:write call. Add a per-session counter incremented on each write, readable via ptyManager.getWriteCount(sessionId). Reset to 0 on session close.

Staleness Thresholds

Both conditions must be true simultaneously to trigger Gate 2:

ThresholdDefaultRationale
staleTimeMs10 minutesLong enough that a quick multi-file fix doesn't trigger it
staleWriteCount50 PTY writesFilters out trivial activity; meaningful work typically exceeds this

TDD Suggestions

  • PlanReadTracker.recordRead() stores record retrievable by getRead
  • PlanReadTracker.isStale() returns false when only time exceeded (writes below threshold)
  • PlanReadTracker.isStale() returns false when only writes exceeded (time below threshold)
  • PlanReadTracker.isStale() returns true when both time AND writes exceed thresholds
  • PlanReadTracker.clear(sessionId) removes all records for that session
  • formatPlanRecap(item, 'never-read') produces Gate 1 block with plan_get instruction
  • formatPlanRecap(item, 'stale', { minutesAgo, writesSince }) produces Gate 2 block with elapsed stats
  • formatPlanRecap(item, 'pass') with AC section → Gate 3 block listing criteria
  • formatPlanRecap(item, 'pass') with Done Statement section → Gate 3 block listing done items
  • formatPlanRecap(item, 'pass') with no sections → Gate 3 soft block with re-read instruction
  • plan-handlers: plan:get calls PlanReadTracker.recordRead with current PTY write count
  • plan-handlers: plan:complete with no prior read → ptyWrite Gate 1, returns error (plan not marked done)
  • plan-handlers: plan:complete with stale read → ptyWrite Gate 2, returns error
  • plan-handlers: plan:complete with fresh read → ptyWrite Gate 3, plan marked done
  • plan-handlers: plan:complete with completionRecap=false → no PTY write, normal completion
  • PtyManager write counter increments on each pty:write; resets on session close

Acceptance Criteria

  • "Completion recap" checkbox in plan edit form, row 2 alongside Auto-implement
  • Tooltip present and accurate
  • Flag persists across saves and restarts
  • Gate 1: agent completes without reading → plan stays unchanged, Gate 1 PTY block written
  • Gate 2: agent read the plan but has been working for >10 min with >50 writes since → plan stays, Gate 2 block written with real elapsed minutes and write count
  • Gate 3: fresh read + AC/Done Statement found → plan marked done, recap block written
  • Gate 3 soft: fresh read + no sections → plan marked done, re-read instruction written
  • recap=OFF → no gate checks, normal completion
  • No sessionId on plan → gate skipped silently
  • 4-row edit layout implemented; existing plans (no flag) behave as recap=OFF
  • Gamepad D-pad cycles through new checkbox; tooltip on focus

Complexity Assessment

Effort: Medium (2–3 days) Risk: Low — additive, flag-gated Touch points: 6 files + 2 new No disk persistence for read tracker — restart clears it (intentional)
