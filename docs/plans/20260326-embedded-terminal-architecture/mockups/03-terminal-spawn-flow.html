🚀 Terminal Spawn Flow
What happens when user presses LT to spawn a new Claude Code terminal
1
🎮 Gamepad Trigger
User presses LT (Left Trigger) — bound to spawn claude-code. Or selects a CLI type from the sidebar spawn panel.
↓
2
⚡ PTY Process Created ~50ms
Main process: PtyManager.spawn('claude-code', cwd) creates a new pseudo-terminal via node-pty. New tab appears in terminal area. State: 💤 IDLE
claude-code #3 — X:\coding\gamepad-cli-hub
Starting Claude Code...
↓
3
🖥️ CLI Initialises ~1-3s (configurable wait)
The CLI tool (claude/copilot/opencode) loads and shows its welcome prompt. The app waits for the configured initialPromptDelay before sending the system prompt.
claude-code #3
╭─────────────────────────────────╮
│ Claude Code v1.2.3 │
│ Working dir: X:\coding\... │
╰─────────────────────────────────╯
claude >
↓
4
📝 System Prompt Pre-Loaded (NOT Submitted) automatic
The app writes the initialPrompt to the PTY's stdin without pressing Enter. The text appears in the CLI's input buffer. The user can review, edit, clear, or submit when ready. State remains: 💤 IDLE
claude-code #3 — 💤 Idle (prompt pre-loaded)
╭─────────────────────────────────╮
│ Claude Code v1.2.3 │
╰─────────────────────────────────╯
claude > <application-prompt>
SYSTEMPROMPT:
- Include AIAGENT-QUESTION when done.
- Include AIAGENT-IMPLEMENTING when coding.
- End responses with an open ended question.
</application-prompt>
⏎ Press Enter to send • Ctrl+C to clear • Edit freely
↓
5
👤 User Decides — Send, Edit, or Clear
The user has full control:
• A / Enter — Submit the pre-loaded prompt as-is (state → 🔨 IMPLEMENTING)
• Edit the text before sending — add project-specific instructions
• Ctrl+C — Clear the prompt entirely and start fresh
The prompt stays in the buffer indefinitely until the user acts.
claude-code #3 — User pressed Enter → 🔨 Implementing
claude > <application-prompt>...</application-prompt>
✓ Understood. I'll include state keywords in all responses.
AIAGENT-QUESTION
What would you like me to work on?
claude >
↓
⚙
📄 Configuration — config/tools.yaml
Each CLI type defines its command and initial prompt. The initialPromptDelay controls how long to wait after the CLI loads before injecting the prompt.
cliTypes:
  claude-code:
    name: Claude Code
    command: claude
    initialPromptDelay: 2000 # ms to wait before sending
    initialPrompt: |
      SYSTEMPROMPT:
      - Include AIAGENT-QUESTION when done with a task.
      - Include AIAGENT-IMPLEMENTING when actively coding.
      - Include AIAGENT-PLANNING when researching.
      - Include AIAGENT-IDLE when awaiting input.
      - End responses with an open ended question.
  copilot-cli:
    name: GitHub Copilot CLI
    command: copilot
    initialPromptDelay: 2000
    initialPrompt: |
      SYSTEMPROMPT: ... # same as above
  opencode:
    name: OpenCode
    command: opencode
    initialPromptDelay: 3000
    initialPrompt: |
      SYSTEMPROMPT: ...

📐 Mockup 03: Spawn Flow + Initial Prompt

  • Spawn trigger — gamepad button (LT/RB) or sidebar spawn panel
  • PTY creation — node-pty.spawn() with piped stdio, not detached
  • Initial prompt — pre-loaded into input buffer, NOT auto-sent
  • User sees prompt wrapped in <application-prompt> tags
  • User decides: Enter to send, edit first, or Ctrl+C to clear
  • AIAGENT-* keywords in CLI output drive automatic state transitions
  • State flow: IDLE → IMPLEMENTING (prompt sent) → WAITING (AIAGENT-QUESTION received)
  • The terminal: wt field is removed — no external terminal launcher needed
  • Each CLI type can have its own initialPromptDelay and initialPrompt
