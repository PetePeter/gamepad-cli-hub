Embedded Terminal Architecture

Transform gamepad-cli-hub from external window management to fullscreen embedded terminals with PTY piping, pipeline-based state management, and auto-handoff between AI agents.

📋 Problem Statement

The current architecture spawns CLI sessions as external terminal windows (Windows Terminal), managing them via Win32 window focus/enumeration and sending keystrokes via robotjs. This is fragile — it depends on OS-level window management, global keystroke injection, and offers no visibility into terminal content or state.

Goal: Transform the app into a fullscreen Electron application that hosts terminals internally using PTY (pseudo-terminal) processes, pipes stdin/stdout/stderr through IPC, and renders them with xterm.js. Each terminal has a state category that's auto-detected via AIAGENT-* keywords or manually overridden.

Data Flow

Gamepad Button Press
  → Navigation routes input
    → If terminal focused: IPC 'pty:write' → Main → PTY stdin
    → If state menu open: Select state transition
    → If sidebar focused: Switch terminal / Spawn new

PTY stdout/stderr
  → Main: PtyManager receives data
    → StateDetector scans for AIAGENT-* keywords
    → IPC 'pty:data' → Renderer → xterm.js terminal.write()
    → If AIAGENT-QUESTION → idle + auto-handoff next waiting

🔄 Pipeline Workflow

The states form a pipeline: planning → waiting → implementing → idle. The app manages transitions both manually (user-driven via state menu) and automatically (AIAGENT-* keywords + auto-handoff).

🔨 Implementing Green dot (pulsing) Auto (from waiting queue) or manual CLI is actively coding/writing
⏳ Waiting Amber dot (pulsing) Manual (user queues a planned session) Plan ready; queued to auto-start when a slot opens
🧠 Planning Blue dot Manual or AIAGENT-PLANNING keyword CLI is researching/thinking; NOT ready to implement
💤 Idle Grey dot AIAGENT-IDLE keyword or no output timeout No active task

Question Badge [?]

When a terminal emits AIAGENT-QUESTION, a [?] badge appears on its tab header. This does not change the terminal's state — an implementing terminal with a question is still implementing. The badge clears automatically when the terminal emits more output (the agent continues).

Key: AIAGENT-QUESTION ≠ idle. It's a notification flag, not a state transition. The user can see at a glance which terminals have pending questions without interrupting the workflow.

Auto-Handoff Logic

When an implementing session emits AIAGENT-IDLE (task complete):

  1. That session transitions to 💤 Idle
  2. App checks the waiting queue (FIFO, ordered by time entered waiting)
  3. If a waiting session exists: app writes "go implement it" + Enter to its PTY
  4. That session transitions to 🔨 Implementing
  5. If no waiting sessions: nothing happens, all stay idle
Worker Pool Pattern: Plan across multiple sessions, queue them up via state menu, and the app auto-dispatches work as implementers finish. No manual babysitting required.

State Menu (Gamepad Y Button)

Pressing Y on the active terminal opens a modal overlay with 4 options:

OptionEffect
⏳ Move to WaitingQueue for implementation — auto-starts when a slot opens
🔨 Force ImplementSkip queue — start implementing immediately
🧠 Move to PlanningMark as researching/thinking
💤 Move to IdlePark this session — no auto-actions

📨 Initial Prompt — Per CLI Type

On each terminal spawn, after the CLI loads, the system prompt is written into the PTY input buffer but NOT submitted. The user sees it pre-filled and decides when to send, edit, or clear.

Key: Each CLI type has its own initial prompt and delay. Wrapped in <application-prompt> tags. The user has full control — the prompt just sits in the input buffer waiting. 
cliTypes:
  claude-code:
    name: Claude Code
    command: claude
    initialPromptDelay: 2000   # ms to wait before pre-loading
    initialPrompt: |
      <application-prompt>
      SYSTEMPROMPT:
      - Include AIAGENT-QUESTION when done with a task.
      - Include AIAGENT-IMPLEMENTING when actively coding.
      - Include AIAGENT-PLANNING when researching.
      - Include AIAGENT-IDLE when awaiting input.
      - End responses with an open ended question.
      </application-prompt>

  copilot-cli:
    name: GitHub Copilot CLI
    command: copilot
    initialPromptDelay: 2000
    initialPrompt: |
      # Same AIAGENT-* protocol, different per-CLI customisations

  opencode:
    name: OpenCode
    command: opencode
    initialPromptDelay: 3000

  generic-terminal:
    initialPromptDelay: 0     # no prompt for generic terminals
    initialPrompt: ""

⌨️ Sequence Parser

A sequence string field on keyboard bindings replaces the old robotjs-based keys array. All input now goes through PTY stdin. The same parser handles initial prompt pre-loading.

SyntaxMeaningExample
plain textType each character to PTY/clear
{Key}Tap a special key{Enter}, {Tab}
{Mod+Key}Combo (hold mod, tap key){Ctrl+S}
{Mod Down}Press & hold modifier{Ctrl Down}
{Mod Up}Release modifier{Ctrl Up}
{Wait N}Pause N milliseconds{Wait 500}
{{ / }}Literal braces{{json}}
newline (YAML |)Press EnterMulti-line blocks

Dual Purpose

  1. Gamepad button bindings: sequence field → parse → write to active terminal's PTY
  2. Initial prompt: initialPrompt in tools.yaml → parse → pre-load into PTY (no Enter)
# Old (robotjs, global keystrokes):
A:
  action: keyboard
  keys: [/, c, l, e, a, r]

# New (PTY stdin, rich format):
A:
  action: keyboard
  sequence: |
    /clear
