Helm User Guide

Xbox controller → CLI session manager for AI coding terminals

1. Getting Started

Helm is a desktop app that lets you control multiple AI coding CLIs (Claude Code, Copilot CLI, etc.) from a single gamepad. Terminals are embedded — no external windows.

First launch

  • Connect your Xbox controller (USB or Bluetooth)
  • Launch Helm — you'll see the session screen with the quick-spawn grid
  • Press D-Pad Up/Down to highlight a CLI tool, then A to spawn a session
  • Start coding with your controller as the input device
Generic / DirectInput gamepads are also supported. D-pad is detected via axes or hat switch automatically.

4. Sessions

Spawning sessions

  • Use the quick-spawn grid at the bottom of the session screen
  • Navigate with D-Pad, press A to spawn
  • Each CLI tool is a separate card in the grid

Session cards

Each session card displays:

  • Activity dot — indicates PTY I/O status:
    • Active (green) — producing output or receiving input
    • Inactive (blue) — silent for >10 seconds
    • Idle (grey) — silent for >5 minutes
  • Elapsed timer — time since last CLI output, refreshed every 10 seconds
  • Plan badges — coding + ready/blocked/review counts
  • Draft badge — count of unsent drafts

Groups

  • Sessions are grouped by working directory with collapsible headers
  • Group order and collapse state persist across restarts
  • Bookmarked directories appear as headers even with 0 active sessions (auto-bookmarked when a named session is removed)

Group overview

  • Press D-Pad → on a group header to open a scrollable preview grid
  • Each card shows: session name, terminal title, activity dot, last 10 lines of output
  • Navigate with D-Pad, press A to select, B to exit

5. Notifications

  • Bubble notifications — displayed when the AI agent uses notify_user
  • Notification carousel — per-session card, shows recent notifications
  • Clearing — press Ctrl+Shift+B to dismiss all notifications for the active session
  • Telegram mirror — PTY output is buffered during active periods and flushed to Telegram on activity change, question detection, or safety threshold (50 KB)
Telegram mirroring is activity-gated: only output produced during green-dot (active) periods is mirrored. Content is ANSI-stripped and noise-filtered before sending.

7. Plans

Plans are per-directory work items organized as an acyclic directed graph (DAG).

Lifecycle

planning→ ready→ coding→ review→ done | blocked
  • Planning — not ready or still being shaped
  • Ready — dependencies satisfied, available to claim
  • Coding — claimed by a session
  • Review — awaiting review
  • Blocked — requires resolution (see stateInfo)
  • Done — completed

Using plans

  • Open: click the &127758; button on a group header
  • Canvas shows nodes with dependency arrows, pan/zoom via mouse
  • Click a node to select it; editor panel at the bottom for title, description, and actions
  • Session cards display plan badges (coding + ready/blocked/review counts)
  • Claim a ready plan from the draft strip — click the ready chip to send its description to the terminal and transition to coding

Backup & restore

  • Press R or click the &128190; button in the plan screen to view backups
  • Automatic rolling-window snapshots on plan changes
  • Configurable interval and retention in Settings → Backups

8. Settings & Profiles

  • Open: click the &9881; button
  • Slides in as an overlay panel

Profiles

Each profile is a single YAML file containing:

  • Tools — CLI definitions, spawn commands, working directories
  • Directories — configured working directories
  • Bindings — per-CLI button mappings (keyboard, voice, sequence, spawn)
  • Patterns — regex rules for auto-actions
  • Stick / D-pad config — dead zones, sensitivity, mode

Switching profiles changes all bindings, tools, and patterns at once. Legacy tools.yaml / directories.yaml files are auto-migrated on first load.

9. Pattern Matcher

The pattern matcher scans every PTY output chunk against per-CLI regex rules defined in the profile.

Actions

TypeBehavior
send-textFires a sequence to the terminal immediately on match
wait-untilParses a scheduled time from the matched capture group, then fires the resume sequence at that time

Schedule management

  • When a wait-until rule fires, a &9200; chip appears on the session card
  • Cancelling: use the × button on the chip or pattern:cancelSchedule
  • Per-rule cooldown prevents rapid re-triggering
