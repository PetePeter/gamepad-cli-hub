🗺️ Directory Plans — Connected Diagram (NCN)

Date: 2026-04-14  |  Status: Draft — awaiting approval

Mission: Add a per-directory DAG of work items ("plans") to the gamepad-cli-hub. Users create plan nodes with title + description, draw dependency arrows between them, and the system tracks a 4-state lifecycle: pending → startable → doing → done. Startable plans appear as green chips on session cards, allowing users to apply them to terminals. Inspired by WiseTech Global's "NCN" (Network Connected Nodes).

Data Model

PlanItem

interface PlanItem {
  id: string;                // UUID v4
  dirPath: string;           // working directory this plan belongs to
  title: string;             // short label (displayed on canvas node)
  description: string;       // longer body (displayed in editor)
  status: PlanStatus;        // 'pending' | 'startable' | 'doing' | 'done'
  sessionId?: string;        // set when status = 'doing' (which session picked it up)
  createdAt: number;         // timestamp
  updatedAt: number;         // timestamp
}

type PlanStatus = 'pending' | 'startable' | 'doing' | 'done';

PlanDependency

interface PlanDependency {
  fromId: string;   // blocker (must be done first)
  toId: string;     // blocked (cannot start until fromId is done)
}
// Arrow: fromId → toId = "fromId blocks toId"
// A → B means A must finish before B can start

DirectoryPlan (aggregate per directory)

interface DirectoryPlan {
  dirPath: string;
  items: PlanItem[];
  dependencies: PlanDependency[];
}

Key Design Decisions

📐 dag-builder-js for canvas rendering
Lightweight (~70KB) SVG-based DAG library with built-in cycle prevention, pan/zoom, edge drag-creation, and import/export. No React dependency. We'll wrap it with our own auto-layout algorithm since it only provides manual positioning. Nodes are rendered as SVG shapes with HTML overlays for title + status dot.
📐 Auto-layout only (left-to-right)
No manual node positioning. We compute a layered DAG layout (topological sort → layer assignment → vertical ordering within layers) and set coordinates. Re-layout triggers on every node/edge add/remove. Direction: roots on left, leaves on right.
📐 Plans are separate from drafts
Plans = folder-level strategic work items with dependencies. Drafts = session-level tactical prompt memos. Startable plans appear as chips in the session's draft-chip area but are a distinct concept. A plan chip opens a draft-style editor (not the draft editor itself).
📐 No cross-directory dependencies
Each directory's plan graph is independent. Data model uses globally unique IDs (UUID) to allow cross-dir in the future, but the DAG validation and startable computation are scoped per-directory.
📐 Keyboard/mouse only on canvas
The plan canvas is not gamepad-navigable. Group header gets a "Plans" button (🗺️) for mouse/keyboard activation. Gamepad users interact with plans only via the session card chips.
📐 Startable auto-computation
When a plan item is marked "done", the PlanManager recomputes startable status for all items in that directory. An item becomes startable when ALL its dependencies (incoming edges where it is the toId) have status === 'done'. Items with no dependencies are startable immediately on creation.

Auto-Layout Algorithm

// Left-to-right layered layout
// 1. Topological sort all nodes
// 2. Assign each node to a layer (longest path from any root)
// 3. Within each layer, order nodes to minimise edge crossings (barycenter heuristic)
// 4. Assign x = layer * LAYER_SPACING, y = position_in_layer * NODE_SPACING
// 5. Set coordinates on dag-builder-js Vertex instances
// 6. Call graph.update() to re-render

const LAYER_SPACING = 280;  // horizontal gap between layers
const NODE_SPACING = 140;   // vertical gap between nodes in same layer
const NODE_WIDTH = 220;
const NODE_HEIGHT = 90;

Chip Click Flow

sequenceDiagram
    actor User
    participant Chip as Plan Chip (session card)
    participant Editor as Plan Editor Modal
    participant IPC as IPC Bridge
    participant PM as PlanManager
    participant PTY as PTY (terminal)

    User->>Chip: Click green plan chip
    Chip->>IPC: plan:get(planId)
    IPC->>PM: getItem(id)
    PM-->>Editor: PlanItem data
    Editor->>User: Show editor (title + description)

    alt Apply
        User->>Editor: Click "▶️ Apply"
        Editor->>PTY: ptyWrite(description)
        Editor->>IPC: plan:apply(id, sessionId)
        IPC->>PM: applyItem(id, sessionId)
        PM-->>Chip: status → doing (blue chip)
    else Save
        User->>Editor: Edit title/desc → Click "💾 Save"
        Editor->>IPC: plan:update(id, changes)
        IPC->>PM: updateItem(id, changes)
    else Mark Done
        User->>Editor: Click "✅ Mark Done"
        Editor->>IPC: plan:complete(id)
        IPC->>PM: completeItem(id)
        PM->>PM: recomputeStartable(dirPath)
        PM-->>Chip: status → done (chip removed)
    end

Marking Done

Two paths to mark a plan "doing" → "done":

  1. From session chip: Blue "doing" chip on session card → click → editor with "Mark Done" button
  2. From plan canvas: Click the node → editor panel → "Mark Done" button

When marked done:

  • Status set to 'done', sessionId cleared
  • PlanManager recomputes startable status for all dependents
  • Newly startable items appear as green chips on relevant session cards
  • Node on canvas: opacity 0.5, title strikethrough, grey dot

Auto-Layout Algorithm

UI TDD

Implement the left-to-right layered DAG layout algorithm used to position nodes on the canvas.

Deliverables

  • renderer/plans/plan-layout.ts — Layout algorithm:
    • computeLayout(items, dependencies) → Map<string, {x, y}>
    • Step 1: Topological sort (Kahn's algorithm)
    • Step 2: Layer assignment (longest path from roots)
    • Step 3: Within-layer ordering (barycenter heuristic to minimise edge crossings)
    • Step 4: Coordinate assignment (layer × LAYER_SPACING, position × NODE_SPACING)
    • Constants: LAYER_SPACING=280, NODE_SPACING=140, NODE_WIDTH=220, NODE_HEIGHT=90
  • tests/plan-layout.test.ts — Layout algorithm tests:
    • Single node → layer 0, position (0,0)
    • Linear chain A→B→C → three layers, left to right
    • Diamond pattern (A→B, A→C, B→D, C→D) → proper layering
    • Multiple roots → all at layer 0
    • Multiple sinks → all at rightmost layer
    • Edge crossing minimisation (verify ordering)
    • Empty graph → empty result

Risk Register

RiskImpactMitigation
dag-builder-js doesn't support programmatic position updates (only drag) High — auto-layout broken Spike in Phase 4: test setting vertex positions via API. Fallback: destroy+recreate vertices on re-layout. Worst case: switch to raw SVG + dagre.
dag-builder-js lacks TypeScript types Low — inconvenience Write a minimal .d.ts declaration file covering the API surface we use.
Performance with 50 nodes + re-layout on every change Medium — sluggish UX Debounce re-layout (100ms). Profile with 50 nodes early. DOM-based rendering at this scale should be fine.
dag-builder-js ESM compatibility with esbuild Medium — build failure Test import in esbuild early. May need to be --external or use a CDN fallback.
Startable chip refresh latency (IPC round-trip per session card) Low — minor delay Batch IPC: single plan:startableForDir call per directory, distribute to all session cards.
