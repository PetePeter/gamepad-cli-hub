Chip-Bar Action Buttons

Add configurable quick-action buttons to the right end of the chip-bar strip above the terminal. Buttons send predefined prompts (using sequence-parser syntax) to the active PTY. Configured in the profile YAML at the root level — shared across all CLI types. A default 💾 Save Plan button is pre-populated.

Problem Statement

The chip-bar above each terminal currently shows draft pills and plan chips — purely read-only indicators. There is no quick way to inject a canned prompt into the active CLI without opening a context menu or navigating to a sequence-picker.

The immediate need: a button that tells the active CLI to record a plan item in the format the hub app expects — pointing to the right config directory, using the correct JSON schema — without the user having to compose and type that every time.

The solution should be profile-configurable (YAML), not hardcoded, so the user can change the prompt text and add more buttons over time.

Solution 1 — Profile-Root chipActions, Inline Render Recommended

Overview

Add a chipActions array to the profile YAML at the root level (not per-tool). Each entry has a label and a sequence using the existing sequence-parser syntax. The chip-bar renders these as small buttons on the right, separated by a visual divider. Clicking a button sends the sequence to the active PTY via ptyWrite().

Data flow

flowchart LR YAML[default.yaml\nchipActions array] --> CL[ConfigLoader\ngetChipbarActions] CL --> IPC[config:getChipbarActions\nIPC handler] IPC --> PL[preload\nconfigGetChipbarActions] PL --> DS[draft-strip.ts\nrenderActionButtons] DS --> DOM[chip-action-bar div\nright-aligned buttons] DOM -->|click| SP[sequence-parser\nparse sequence] SP --> PTY[ptyWrite → active PTY]

Profile YAML shape

# config/profiles/default.yaml (root level, alongside tools/bindings)
chipActions:
  - label: "💾 Save Plan"
    sequence: "Please save the current task as a plan item.\nStore it at config/plans/ using a UUID filename.\nJSON fields: id (uuid-v4), dirPath (current working dir), title (short), description (full detail), status (startable), createdAt/updatedAt (unix ms).\nExample filename: X%3A%5Ccoding%5Cproject@{uuid}.json{Enter}"
  - label: "📋 Status"
    sequence: "/status{Enter}"

Rendering in draft-strip.ts

A dedicated renderActionButtons() function appends (or re-uses) a right-aligned .chip-action-bar div at the end of the strip. It fetches the config once and caches it, re-fetching on profile switch. Called at the end of refreshDraftStrip().

// draft-strip.ts additions

let cachedActions: ChipbarAction[] | null = null;

async function getActions(): Promise<ChipbarAction[]> {
  if (!cachedActions) cachedActions = await window.gamepadCli.configGetChipbarActions();
  return cachedActions;
}

export function invalidateActionsCache(): void { cachedActions = null; }

async function renderActionButtons(sessionId: string): Promise<void> {
  const strip = document.getElementById('draftStrip');
  if (!strip || !sessionId) return;

  strip.querySelector('.chip-action-bar')?.remove();

  const actions = await getActions();
  if (actions.length === 0) return;

  const bar = document.createElement('div');
  bar.className = 'chip-action-bar';

  for (const action of actions) {
    const btn = document.createElement('button');
    btn.className = 'chip-action-btn';
    btn.textContent = action.label;
    btn.title = action.sequence.slice(0, 80);
    btn.addEventListener('click', async () => {
      const { parseSequence, executeSequence } = await import('../input/sequence-runner.js');
      await executeSequence(parseSequence(action.sequence), sessionId);
    });
    bar.appendChild(btn);
  }

  strip.appendChild(bar);
  strip.style.display = 'flex';
}

Visibility rule

Action buttons are part of the strip and therefore share the same visibility: the strip is only shown (display: flex) when it has content (drafts, plan chips, or action buttons). Since action buttons are appended unconditionally when a session is active, the strip will always be visible for active sessions — which is the intended behaviour.

Test proposals

  • Strip rendering: After refreshDraftStrip(sessionId) with two configured actions, a .chip-action-bar with two buttons appears at the end of the strip.
  • Click dispatches sequence: Mock executeSequence — clicking button calls it with the correct sequence string and sessionId.
  • Cache invalidation: After invalidateActionsCache(), the next render re-fetches from IPC.
  • Empty actions: No .chip-action-bar rendered when chipActions: [].
  • Session null: No crash when sessionId is null — buttons not rendered.

Solution 2 — Template Variables in Sequence Extension

Overview

Identical to Solution 1, but extends the sequence string with context-aware placeholders resolved at click time. This allows the "Save Plan" prompt to include the actual working directory path, CLI type, or session name without the user needing to type them.

This is an additive extension on top of Solution 1 — not a replacement. Implement Solution 1 first, then layer this on.

Template variables

PlaceholderResolved value
{cwd}Working directory of the active session
{cliType}CLI type key (e.g. claude-code)
{sessionName}Display name of the session
{plansDir}Absolute path to config/plans/

Example YAML with template

chipActions:
  - label: "💾 Save Plan"
    sequence: >-
      Please save a plan item for the current task to {plansDir}.
      Working directory: {cwd}.
      Use JSON: { "id": "<uuid-v4>", "dirPath": "{cwd}", "title": "...",
      "description": "...", "status": "startable",
      "createdAt": <unix-ms>, "updatedAt": <unix-ms> }{Enter}

Resolution logic

function resolveTemplates(sequence: string, ctx: ActionContext): string {
  return sequence
    .replace(/\{cwd\}/g, ctx.cwd ?? '')
    .replace(/\{cliType\}/g, ctx.cliType ?? '')
    .replace(/\{sessionName\}/g, ctx.sessionName ?? '')
    .replace(/\{plansDir\}/g, ctx.plansDir ?? '');
}

ActionContext is built from state.sessions at click time — no async lookup needed if the session state is already cached.

Test proposals

  • Template resolution: resolveTemplates('{cwd}/plans', { cwd: 'X:\\proj', ... }) returns 'X:\\proj/plans'.
  • Click uses resolved sequence: Template vars are substituted before passing to executeSequence.
  • Missing vars default to empty string: No crash when context fields are undefined.

Conclusion

Recommendation: implement Solution 1 now, optionally layer Solution 2 (template variables) immediately after since it adds minimal complexity but makes the "Save Plan" prompt genuinely useful (embeds the actual working directory into the prompt text).

Solution 3 (Settings UI) is future scope — validate the YAML workflow first.

The implementation is contained to ~5 files, reuses existing IPC patterns, and requires no new architecture. The default chipActions entry gives the user a working "Save Plan" button out of the box, and the YAML-driven design means they can add more without any code changes.

File count estimate: 5 source files modified + 1 new test file + 1 YAML update. No new IPC architecture — just one new handle() call.
