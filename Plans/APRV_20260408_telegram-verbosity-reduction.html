Telegram Verbosity Reduction

Goal: Replace real-time terminal mirroring with activity-gated batching — send the LLM's settled output to Telegram only after the session goes quiet (blue activity dot), instead of streaming every chunk in real-time.

User requirements (confirmed via Q&A):
  • Wait for the LLM to go idle (blue activity dot, >10s silence) before sending to Telegram
  • Send the final settled terminal content, not intermediate streaming/thinking/progress
  • One global "smart" mode — no per-session toggles needed
  • Fewer, larger messages instead of constant edit-in-place
  • No progress indicators, repeated text, or thinking animations
  • Full stream still visible in-app via xterm.js — Telegram gets the digest

Problem Statement

The TerminalMirror module currently mirrors all PTY output to Telegram in near real-time (1.5s–3s debounced flushes). AI CLIs like Claude Code and Copilot CLI produce extremely verbose streaming output:

  • Thinking/progress indicators — spinners, "thinking..." animations, percentage counters
  • Streaming tokens — LLM responses arrive character-by-character, causing constant message edits
  • Repeated/overwritten content — terminal carriage returns cause the same text to appear multiple times in the mirror
  • Tool call output — verbose build/test/lint output that's useful in-app but noise on Telegram

Proposed Solution 1 — Activity-Gated Batching RECOMMENDED

Replace TerminalMirror's real-time flushing with activity-gated batching: buffer output while the session is active (green dot), flush only when the session transitions to inactive (blue dot, >10s silence). Additionally, echo user prompts to Telegram immediately when submitted from the in-app terminal.

Key Changes

1. src/telegram/terminal-mirror.ts

  • Remove timer-based scheduleFlush() — no more 1.5s/3s debounced flushes
  • feedOutput() only appends to buffer (with safety cap at 100KB)
  • Add handleActivityChange(sessionId, level):
    • 'inactive' → flush buffer as a new message (not edit-in-place)
    • 'idle' → flush if buffer has content (catch stragglers)
    • 'active' → no-op (just buffering)
  • Each flush sends a new message per activity burst — no edit-in-place pattern
  • Add safety flush: if buffer exceeds MAX_BUFFER_CHARS (e.g., 50KB), flush immediately regardless of activity state (prevents memory pressure during long-running builds)
  • Add trackInput(sessionId, data) — accumulates typed characters per session, sends prompt to Telegram on Enter (\r)

Prompt Echo — Input Tracking

A per-session inputBuffer accumulates text from pty:write events (in-app only — Telegram topic input bypasses IPC). When \r is detected:

  1. Extract printable text from the buffer (strip escape sequences, control chars)
  2. If text is non-empty → send 📝 <text> to the session's Telegram topic
  3. Clear the input buffer

This creates a natural conversation flow: prompt message → (silence while LLM works) → settled response message.

// Input tracking
trackInput(sessionId: string, data: string): void {
  const state = this.getOrCreateState(sessionId);
  state.inputBuffer += data;

  if (data.includes('\r') || data.includes('\n')) {
    const text = stripControlChars(state.inputBuffer).trim();
    state.inputBuffer = '';
    if (text.length === 0) return;  // Enter-only, Ctrl+C, arrows → skip

    const topicId = this.topicManager.getTopicId(sessionId);
    if (!topicId) return;
    const formatted = `📝 ${escapeHtml(text)}`;
    this.bot.sendToTopic(topicId, formatted, { parse_mode: 'HTML' });
  }
}

// Output buffering (no immediate flush)
feedOutput(sessionId: string, data: string): void {
  const state = this.getOrCreateState(sessionId);
  state.buffer += data;

  // Safety flush: don't let buffer grow unbounded
  if (state.buffer.length > MAX_BUFFER_CHARS) {
    this.flush(sessionId);
  }
}

handleActivityChange(sessionId: string, level: ActivityLevel): void {
  if (level === 'inactive' || level === 'idle') {
    this.flush(sessionId);
  }
}

private async flush(sessionId: string): Promise<void> {
  const state = this.states.get(sessionId);
  if (!state || !state.buffer.trim()) return;

  const cleaned = cleanTerminalOutput(state.buffer);
  state.buffer = '';
  if (!cleaned.trim()) return;

  const content = fitToLimit(cleaned);
  const formatted = `<code>${escapeHtml(content)}</code>`;

  // Always send NEW message — no edit-in-place
  await this.bot.sendToTopic(state.topicId, formatted, { parse_mode: 'HTML' });
}

Pros / Cons

  • ✅ Leverages existing StateDetector infrastructure — no duplicated timing logic
  • ✅ Naturally produces fewer, larger messages (1–2 per LLM turn)
  • ✅ Content is genuinely settled by the time it's sent
  • ✅ cleanTerminalOutput() collapses carriage-return overwrites, eliminating duplicates
  • ✅ Safety flush prevents memory issues during long builds
  • ✅ Removes the edit-in-place complexity (no more editMessageDebounced)
  • ❌ 10s delay before any output appears on Telegram — user sees nothing during active work
  • ❌ Requires wiring a new callback through handlers.ts → pty-handlers.ts → orchestrator
  • ❌ Very long active periods (e.g., 10min build) may trigger safety flush mid-stream

Complexity

Medium. ~5 files changed, removes complexity from TerminalMirror (no timers, no edit-in-place), adds one new callback wire. The hardest part is testing the activity-trigger flow.

Proposed Solution 2 — Quiet Period Timer ALTERNATIVE

Instead of wiring to StateDetector events, change TerminalMirror's existing flush timer from 1.5s to a configurable quiet period (default 10s). Each new output chunk resets the timer. Content only flushes after N seconds of silence.

Key Changes

src/telegram/terminal-mirror.ts only

  • Change NORMAL_FLUSH_MS from 1500 → 10000 (or configurable)
  • Change scheduleFlush() to reset the timer on every new chunk (current code skips if timer exists)
  • Keep edit-in-place pattern OR switch to new-message-per-flush
// Current: scheduleFlush skips if timer exists
private scheduleFlush(sessionId: string, state: MirrorState): void {
  if (state.flushTimer) return;  // ← skip
  state.flushTimer = setTimeout(() => this.flush(sessionId), interval);
}

// Changed: reset timer on every new chunk
private scheduleFlush(sessionId: string, state: MirrorState): void {
  if (state.flushTimer) clearTimeout(state.flushTimer);  // ← reset
  state.flushTimer = setTimeout(() => this.flush(sessionId), QUIET_PERIOD_MS);
}

Pros / Cons

  • ✅ Minimal change — only TerminalMirror modified (1 file)
  • ✅ No new wiring needed — self-contained within existing module
  • ✅ Same effect as Solution 1 for most scenarios
  • ❌ Duplicates StateDetector's timing logic — two independent 10s timers
  • ❌ If quiet period doesn't match StateDetector threshold, activity dots and Telegram flush can be out of sync
  • ❌ Can't distinguish "paused to read user's question" (should flush) from "paused between tool calls" (might resume)
  • ❌ Timer reset on every chunk means a slow trickle of output (1 byte/9s) could delay flush indefinitely

Complexity

Low. ~1 file, ~10 lines changed. But the duplicated timing logic is a maintenance concern.

Core activity-gated behavior

  1. feedOutput does not flush immediately — only buffers
  2. handleActivityChange('inactive') triggers flush of buffered content
  3. handleActivityChange('idle') triggers flush of buffered content
  4. handleActivityChange('active') does not trigger flush
  5. flush sends a new message (not edit-in-place)
  6. flush cleans content via cleanTerminalOutput before sending
  7. flush does nothing when buffer is empty or whitespace-only
  8. multiple feedOutput calls accumulate in buffer before flush

Safety flush

  1. safety flush triggers when buffer exceeds MAX_BUFFER_CHARS
  2. safety flush resets buffer and sends content

Message lifecycle

  1. each activity burst produces a separate message (no edit-in-place)
  2. rapid active→inactive→active→inactive sends two messages
  3. content is truncated via fitToLimit when exceeding MAX_MESSAGE_CHARS

Content quality

  1. ANSI escape codes are stripped from flushed content
  2. carriage-return overwrites are collapsed (no duplicates)
  3. noise patterns (spinners, AIAGENT tags) are removed
  4. HTML entities are escaped in the Telegram message

Edge cases

  1. removeSession clears buffer, inputBuffer, and state
  2. dispose clears all session states and pending flushes
  3. feedOutput with no topic ID is silently ignored
  4. pause/resume still works (buffer during pause, flush on resume+inactive)

Prompt echo (input tracking)

  1. trackInput accumulates characters in inputBuffer
  2. trackInput sends prompt to Telegram on carriage return
  3. trackInput strips escape sequences and control chars from prompt
  4. trackInput skips empty prompts (Enter-only, Ctrl+C, arrow keys)
  5. trackInput clears inputBuffer after sending
  6. trackInput does nothing when no topic ID exists
  7. trackInput formats message as "📝 prompt text"

Integration wiring (can be unit tested with mocks)

  2. orchestrator.handleActivityChange is no-op when bot not running
  3. orchestrator.trackInput forwards to terminalMirror

Solution 1 directly implements the user's stated preference: "wait for the LLM to be idle (blue activity dot) before messaging Telegram." It leverages the existing StateDetector activity-change events, produces fewer and cleaner messages, and actually simplifies the TerminalMirror code by removing the edit-in-place pattern and debounce timers.

The 10s delay before content appears on Telegram is the desired behavior — the user explicitly doesn't want to see intermediate progress. The safety flush at 50KB provides a reasonable guard for long-running builds.
