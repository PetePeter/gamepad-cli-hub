Fix Folder Name Inconsistency

Ensure custom folder names from the profile config are displayed everywhere a directory group is named — session list headers, plan screen header, and global overview break marks.

Problem Statement

The app has three distinct places that show a folder/directory name. They each derive the name differently:

LocationWhat it showsSource
Session list group headers Path tail — e.g. gamepad-cli-hub dirDisplayName(dir) in groupSessionsByDirectory()
Folder Planner grid buttons Custom name — e.g. Helm dir.name from configGetWorkingDirs()
Plan screen header Path tail — e.g. gamepad-cli-hub folderName(dirPath) local helper
Global overview break marks Full path — e.g. X:\coding\gamepad-cli-hub Raw dirPath directly
Load order constraint: In loadSessions(), groupSessionsByDirectory() is called at line 559, but sessionsState.directories is not populated until lines 566–568 (async, after the group build). A build-time fix would require reordering or a two-phase render.

Proposed Solution 1 — Render-Time Lookup Recommended

Add a pure helper resolveGroupDisplayName(dirPath, directories) to session-groups.ts. Every render site calls this helper with sessionsState.directories at the moment it draws the name. No load-order changes needed.

Respects your choice: render-time lookup, no load-order coupling, pure function that is trivial to test.

New helper (session-groups.ts)

/**
 * Resolve the best display name for a directory.
 * Prefers the custom name from the loaded config; falls back to path tail.
 */
export function resolveGroupDisplayName(
  dirPath: string,
  directories: Array<{ name: string; path: string }>,
): string {
  return directories.find(d => d.path === dirPath)?.name ?? dirDisplayName(dirPath);
}

SessionGroup interface rename

export interface SessionGroup {
  dirPath: string;
  displayName: string;   // renamed from dirName; still stores path-tail initially

Proposed Solution 2 — Build-Time Resolution Alternative

Load directories before calling groupSessionsByDirectory(), then pass them in. The groups come out with fully-resolved names — render sites use group.displayName directly.

Load order change in loadSessions()

// 1. Load directories FIRST
sessionsState.directories = (await window.gamepadCli.configGetWorkingDirs()) || [];

// 2. Then build groups with the directories available
sessionsState.groups = groupSessionsByDirectory(
  state.sessions,
  getSessionCwd,
  sessionsState.groupPrefs,
  sessionsState.directories,   // NEW parameter
);

Updated groupSessionsByDirectory signature

export function groupSessionsByDirectory(
  sessions: Session[],
  getDir: (id: string) => string,
  prefs: SessionGroupPrefs = { order: [], collapsed: [] },
  directories: Array<{ name: string; path: string }> = [],
): SessionGroup[] {
  // ... existing logic ...
  const dirNameMap = new Map(directories.map(d => [d.path, d.name]));
  return orderedDirs.map(dir => ({
    dirPath: dir,
    displayName: dirNameMap.get(dir) ?? dirDisplayName(dir),
    // ...
  }));
}
Cons: Render sites are cleaner (group.displayName direct), but any time the group list is rebuilt before directories are loaded (e.g. initial fast render), the path-tail name shows briefly. Render-time lookup (Solution 1) is always correct.

Proposed Solution 3 — Shared Name Cache Not Recommended

Extract a small module renderer/directory-names.ts that holds a Map<string, string> of dirPath → customName, updated whenever configGetWorkingDirs() resolves. All sites import from this module.

// renderer/directory-names.ts
const _cache = new Map<string, string>();

export function updateDirectoryNames(dirs: Array<{ name: string; path: string }>): void {
  _cache.clear();
  dirs.forEach(d => _cache.set(d.path, d.name));
}

export function getDirectoryDisplayName(dirPath: string): string {
  return _cache.get(dirPath) ?? dirDisplayName(dirPath);
}
Verdict: Adds a mutable global singleton. The same goal is achieved by Solution 1's pure helper passed sessionsState.directories. More complexity for no benefit. Not recommended.

Conclusion

Solution 1 (Render-Time Lookup) is the right choice here. It:

  • Requires no load-order changes in loadSessions()
  • Introduces a single pure, testable helper — resolveGroupDisplayName()
  • Is always correct: uses the freshest sessionsState.directories at the moment each element renders
  • Fixes all four inconsistent display sites: session headers, plan screen title, overview break marks
  • Renames SessionGroup.dirName → displayName for clearer semantics

Total files changed: ~6. Test additions: ~10 new assertions across 3–4 test files.
